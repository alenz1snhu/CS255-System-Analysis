Project Summary

The DriverPass project involved designing a system for a client named DriverPass, a company that provides training to help students pass their driving tests. The client wanted a web-based system that would allow customers to register, purchase training packages, schedule driving lessons, and take practice tests online. They also needed administrative features so staff could manage customers, track lesson schedules, and maintain training content. The goal was to create a system that improved customer access to services while helping the company operate more efficiently.

What I Did Well

One area I did particularly well was translating the client’s business needs into clear system requirements and diagrams. I created multiple UML diagrams, including use case, activity, sequence, and class diagrams, which helped show how the system would function. These diagrams helped connect the business requirements to the technical design and demonstrated how different parts of the system interact. I also focused on making sure the system design supported scheduling, payments, and user management, which were key features requested by the client.

What I Would Improve

If I could revise one part of my work, I would improve the class diagram by refining relationships and including more detailed attributes and methods earlier in the design process. While my diagram met the requirements, improving clarity and organization would make it easier for developers to implement the system. In the future, I would also review the diagrams more thoroughly to ensure they are as clear and easy to understand as possible.

How I Interpreted and Implemented User Needs

I interpreted the user’s needs by reviewing the client’s goals and identifying the key tasks users needed to perform, such as registering, scheduling lessons, and making payments. I then translated those needs into system requirements and UML diagrams that showed how the system would support those tasks. Considering the user’s needs is important because the system must be easy to use, functional, and reliable. If the system does not meet the user’s needs, it will not be effective, no matter how well it is designed technically.

My Approach to Software Design

When designing software, I focus on understanding the problem first and identifying what the user needs the system to do. I use tools such as UML diagrams, requirements documents, and process modeling to visualize how the system should work. These tools help ensure that the design is clear and organized before development begins.

In the future, I will continue using techniques such as:

Requirements analysis

UML diagrams (use case, activity, sequence, and class diagrams)

User-focused design

Documentation and planning

These strategies help create systems that are well-organized, efficient, and meet the needs of the client and users.
